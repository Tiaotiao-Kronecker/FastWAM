ZipMotion / ZipMo 与 FastWAM 集成方案

本文把之前关于 ZipMotion 和 FastWAM 的讨论整理成可执行的技术方案。核心目标不是只在 Bridge、DROID、LIBERO 或 RoboTwin 数据集内提升， 而是让 motion embedding 真正帮助下游机械臂策略，最终能迁移到真机。

无 CDN / 无外部脚本 适合 VS Code Live Preview 日期：2026-05-14 来源：docs/qa 与 docs/plan 归档

1. 目标与当前结论

我们要解决的问题

ZipMotion / ZipMo 擅长从点追踪轨迹中学习 long-term motion embedding。 FastWAM 擅长在当前图像、语言和 proprio 条件下生成 action chunk。 集成目标是让 FastWAM 的 action expert 能读取 ZipMotion 预测的 future-motion tokens，从而改善动作生成。

当前推荐路线

第一版优先做 Context Token Adapter：把 ZipMotion motion tokens 投影到 FastWAM 的 context 维度， 拼到 text/proprio context 后面，让 action/video expert 通过已有 cross attention 读取。

最小可行方案：先离线预计算或 mock 出 motion_tokens，增加 motion_proj: Linear(d_z, 4096)， 在 FastWAM.build_inputs 和 infer_action 中 append motion context，保持原 action flow loss 不变。 
核心数据流：

当前/历史观测
  -> ZipMotion causal predictor
  -> predicted motion tokens
  -> motion_proj 到 4096 维
  -> concat(text_context, proprio_context, motion_context)
  -> FastWAM action expert cross-attention
  -> action flow prediction
  -> action chunk

2. 关键原则

不能泄漏未来

未来真实 tracks 可以作为训练 target，但不能作为 policy input。训练和推理输入必须都来自当前或历史可观测信息。

核心表示要泛化

不建议把 E_DROID、E_Bridge、E_LIBERO 这类 dataset id 放进核心 motion reasoning。

先小改动验证

第一版不要重写 MoT，也不要替换 FastWAM 视频输入。先把 ZipMotion 当额外 context 或 adapter 接入。

推荐使用的可部署条件

camera role: external / wrist / overhead camera intrinsics / extrinsics control frequency action frame convention robot / gripper descriptor proprio schema mask
per-dataset adapter 可以存在，但应放在输入输出边界，用来处理 action space、proprio schema、相机格式和控制频率差异。 不要让核心策略依赖某个 dataset id 才能工作。

3. ZipMotion 与 FastWAM 的表示差异

维度 ZipMotion / ZipMo FastWAM 集成含义
核心对象 start frame + 2D point tracks 拼接视频画布 + VAE latent + action tokens 不要把 ZipMotion 当普通 image encoder，它的强项是点轨迹 motion latent。
多相机 每个 camera view 有自己的 image plane 和 tracks 当前 RobotWin 路径把多相机像素拼成一张画布 ZipMotion 融合应在 per-view motion latent 层做，不应直接相加 2D 坐标。
策略输入 可输出 predicted future-motion tokens action expert 可 cross-attend 到 context tokens 最自然接入点是 FastWAM 的 context/cross-attention 通路。
训练目标 track reconstruction / future motion prediction / action alignment video/action continuous flow matching 第一版保持 FastWAM action flow loss，ZipMotion 可先冻结。 
ZipMotion 形式：

z = ZipMoVAE.encode(start_frame, tracks_yx, visibility, certainty)
future_tracks = ZipMoDecoder(z, query_points, start_frame)

FastWAM action 形式：

A_sigma = (1 - sigma) * A_0 + sigma * epsilon
target_action = epsilon - A_0
pred_action_flow = FastWAM(video_first_frame, context, A_sigma, timestep)

5. 多相机融合：external 主干，wrist 残差

之前讨论的结论是：external camera 通常更稳定、视野更大，更适合作为主 motion latent； wrist camera 变化快、遮挡和模糊多，但对末端接触、抓取细节有价值，应作为 gated residual correction。

多相机 ZipMotion latent 融合 latent/token 层，不是像素层 
z_ext   = ZipMoVAE.encode(start_ext_frame, tracks_ext, visibility_ext)
z_wrist = ZipMoVAE.encode(start_wrist_frame, tracks_wrist, visibility_wrist)

delta = CrossAttention(query=z_ext, key=z_wrist, value=z_wrist)
gate  = sigmoid(MLP(pool(z_ext), pool(z_wrist), proprio, camera_meta))

z_motion = z_ext + gate * delta
z_ext
external camera 的 motion latent，是主干表示。
z_wrist
wrist camera 的 motion latent，提供局部接触和遮挡恢复信息。
delta
wrist 信息对 external motion latent 的修正项。
gate
控制 wrist residual 的强弱，建议初始化偏小，并配合 view dropout。
z_motion
最终传给 FastWAM 的 predicted motion tokens。
external tracks 和 wrist tracks 不在同一个 image plane，通常没有稳定的一一对应 point identity。 因此不能直接把两个视角的 2D tracks 相加，也不建议做像素级残差。

6. FastWAM 当前可利用的接入点

FastWAM 已经有 context/cross-attention 通路。文本 context 和 proprio token 被拼成 context， 分别送入 video expert 和 action expert 的 text_embedding，再作为 cross-attention 的 key/value。

接入点 源码 含义 ZipMotion 可以怎么接
append proprio context fastwam.py:219-240 把 proprio 投影成一个 4096 维 context token。 照这个模式新增 _append_motion_to_context。
training build_inputs fastwam.py:277-383 读取 video/action/context/proprio 并转到 device。 从 sample 读取 motion_tokens 和 motion_mask。
ActionDiT context projection action_dit.py:226-299 把 4096 维 context 投到 action hidden dim 1024。 motion context 拼进去后不需要改这段。
VideoDiT context projection wan_video_dit.py:509-620 把 4096 维 context 投到 video hidden dim 3072。 如果训练 video 分支，也能读 motion context。
CrossAttention wan_video_dit.py:198-220 expert token 作为 query，context 作为 key/value。 motion tokens 变成 context 后自然被读取。
MoT mask fastwam.py:385-407 控制 video/action token 之间的 self-attention。 方案 A 不需要改 MoT mask。

7. 方案 A：Context Token Adapter

这是第一优先级方案。它把 ZipMotion 输出的 motion tokens 当作 FastWAM context 的新增部分， 最大优点是不用改 MoT 结构、不用改 attention mask、不影响 video KV cache 的基本设计。

数学形式 对应 FastWAM context/cross-attention 通路 
Z_motion in R[B, L_m, d_z]

C_motion = Z_motion W_m + b_m
C_motion in R[B, L_m, 4096]

C_prime = concat(C_text, C_prop, C_motion)
M_prime = concat(M_text, M_prop, M_motion)

Action cross-attention:
Q = W_q X_action
K = W_k C_prime
V = W_v C_prime
Attn = softmax(Q K^T / sqrt(d_h) + mask) V
Z_motion
ZipMotion predictor 输出的 predicted motion tokens。
L_m
motion token 数量，可由轨迹点数、时间压缩方式或 latent token 数决定。
d_z
ZipMotion motion token 原始维度。
W_m, b_m
新增的 motion projector，第一版可以是 nn.Linear(d_z, 4096)。
C_prime
扩展后的 FastWAM context，会被 ActionDiT/VideoDiT 内部 MLP 投到各自 hidden dim。

建议新增模块

src/fastwam/models/wan22/zipmotion_context.py
class MotionContextProjector(nn.Module):
    def __init__(self, motion_dim: int, text_dim: int = 4096):
        super().__init__()
        self.proj = nn.Sequential(
            nn.LayerNorm(motion_dim),
            nn.Linear(motion_dim, text_dim),
        )

    def forward(self, motion_tokens):
        # motion_tokens: [B, L_m, d_z]
        return self.proj(motion_tokens)  # [B, L_m, 4096]

FastWAM 内部新增函数

def _append_motion_to_context(self, context, context_mask, motion_tokens, motion_mask=None):
    if self.motion_proj is None or motion_tokens is None:
        return context, context_mask

    motion_tokens = motion_tokens.to(device=self.device, dtype=context.dtype)
    motion_context = self.motion_proj(motion_tokens)  # [B, L_m, 4096]

    if motion_mask is None:
        motion_mask = torch.ones(
            (context.shape[0], motion_context.shape[1]),
            dtype=torch.bool,
            device=context_mask.device,
        )
    else:
        motion_mask = motion_mask.to(device=context_mask.device, dtype=torch.bool)

    return (
        torch.cat([context, motion_context], dim=1),
        torch.cat([context_mask, motion_mask], dim=1),
    )

第一版训练策略

  • 冻结 ZipMotion predictor，只训练 motion_proj 和 FastWAM action 相关参数。
  • 先不改变 video loss，不改变 MoT attention mask。
  • 保持原 action flow loss，验证 motion context 是否被 action expert 利用。
  • 先用 mock/offline motion tokens 做 smoke，确认 shape、mask、forward/backward 都通。
第一版 loss：

L = L_action_flow

可选第二步：

L = L_action_flow + lambda_motion * L_motion_aux

8. 方案 B：Gated Cross-Attention Injection

如果方案 A 有收益但利用强度不足，再做方案 B。它不是把 motion tokens 混入通用 context， 而是在 action expert 若干层中显式加入 motion cross-attention adapter。

动作层内注入公式 第二阶段，不建议第一版直接做 
R_l = CrossAttention(query=LN(h_action_l), key=z_motion_l, value=z_motion_l)
g_l = sigmoid(MLP(pool(h_action_l), pool(z_motion_l), proprio))

h_action_l = h_action_l + alpha_l * g_l * R_l
h_action_l
第 l 层 action expert token。
R_l
motion adapter 读出的 residual。
g_l
样本相关或 token 相关 gate，控制 motion 信息强度。
alpha_l
可学习标量，建议初始化为 0 或很小，使初始模型近似原 FastWAM。

插入位置选择

位置 优点 代价
ActionDiT block 内 cross-attn 后 语义清楚，motion adapter 是额外条件。 需要改 action expert block，训练和推理路径都要同步。
MoT post-block 后 可以在 video/action mixed attention 后再修正 action token。 需要改 MoT._apply_expert_post_block 或新增 action-only post hook。
只在高层注入 减少开销，降低破坏原模型风险。 可能利用不充分，需要 ablation 注入层数。
方案 B 的关键是 gate 初始化。初始时让 adapter 近似无效，先稳定复现 baseline，再逐步学习 motion correction。

9. 方案 C：Motion Expert 加入 MoT

这是更激进的方案，不建议第一版做。它把 motion tokens 当作第三类 expert token 放进 MoT。 

mixtures = {
    "video": video_expert,
    "action": action_expert,
    "motion": motion_expert,
}

为什么不优先

  • motion expert 需要和 video/action expert 对齐 layer 数、head 数、head dim。
  • 需要重新设计 video/action/motion 三方 attention mask。
  • 推理中的 video KV cache 逻辑要扩展为 video/motion cache 或 action-with-motion cache。
  • 调试复杂度明显高于 Context Token Adapter。

11. 推理路线

推理时不能用未来真实 tracks。必须从当前或历史观测预测 motion tokens，再交给 FastWAM。 

input at real robot time t:
  current / historical camera frames
  current proprio
  task language

causal motion path:
  frames up to t
  -> tracker or online track predictor
  -> ZipMotion predictor
  -> predicted motion_tokens
  -> motion_proj
  -> append to FastWAM context

FastWAM action path:
  current image -> VAE first-frame latent -> video KV cache
  random action noise -> iterative denoising
  action expert cross-attends to text + proprio + motion context
  output action chunk
如果训练时 motion tokens 来自未来 GT tracks，而推理时只能用 predicted motion tokens，训练和部署分布会严重不一致。 因此 Phase 2 的 FastWAM 输入应尽早使用 predicted motion tokens，而不是 oracle motion tokens。

13. 实验、消融与风险

必须做的 ablation

实验 目的 预期观察
FastWAM baseline 确认原模型能力。 作为所有对比基线。
external-only ZipMotion 验证稳定外部视角 motion 是否有帮助。 通常应比 wrist-only 更稳定。
wrist-only ZipMotion 验证局部接触信息价值。 可能对抓取接触阶段有效，但长程稳定性较弱。
external 主干 + wrist residual 验证当前推荐多相机融合。 应优于简单 concat 或平权融合。
Context Token Adapter 验证最小接入方式。 若 motion tokens 有效，action loss 或 rollout success 应改善。
Gated Cross-Attn Adapter 验证更强注入是否进一步提升。 应关注稳定性和过拟合风险。

风险与处理

风险 表现 处理
未来信息泄漏 离线验证很好，真机或 causal eval 崩。 严格区分 predicted motion tokens 和 future GT tracks；GT 只能做 target。
motion tokens 被忽略 加 motion 后指标无变化。 增加 auxiliary loss、motion dropout 对照、或进入方案 B。
wrist 噪声伤害策略 加入 wrist 后不稳定。 external 主干、wrist gated residual、view dropout、gate 正则。
dataset shortcut 数据集内好，跨数据集差。 不用核心 dataset id embedding，使用 camera role 和物理描述。
推理延迟 ZipMotion 在线预测增加 replan latency。 先离线评估收益，再做轻量 predictor、缓存历史 tracks、降低 token 数。
